## 工作记忆条目结构说明

每一条工作记忆记录的是智能体的一次代码尝试循环，通常包含以下信息：

- **analysis（分析）**: 智能体的代码生成LLM综合当前环境状态、自身目标、任务要求等信息后得出的分析语句。这是"为什么要这样做"的推理依据，代码是据此分析而编写的。
- **code（代码/代码尝试）**: 基于上述分析生成并执行的具体代码，以及执行结果（成功或失败及错误信息）。
- **decision（决策）**: 代码生成LLM做出的关于任务执行的决策，可能的类型包括：
  - `execute_code`: 正常执行生成的代码
  - `request_modification`: 请求高层修改当前任务（当中层判断当前步骤无法完成或需要调整时）
  - 其他高层决策如 `modify_step`, `retry_step`, `skip_step`, `discarded_task` 等
- **modification_request（修改请求）**: 当decision为request_modification时，描述请求的具体修改内容和原因。

此外还可能包含：与玩家的交互记录、环境观察、高层战略决策等。

## 压缩规则

你需要将上述"已有摘要"与"新增条目"融合，输出一份**完整的**更新后摘要，该摘要将**全量替换**旧摘要。

### 核心原则
1. **增量更新，而非重写**：尽可能保留已有摘要中的内容和措辞。只在新增条目带来了新的、值得记录的信息时才对摘要做修改或补充。如果新增条目没有带来任何新的有价值信息，可以原样返回已有摘要。
2. **保留关键因果链**：每一次重要的"分析→代码→结果"因果链都要清晰保留，让读者能理解"为什么这样做、做了什么、结果如何"。
3. **保留关键细节不可模糊化**：
   - 精确坐标（如 x=-78, y=49, z=-17）
   - 具体物品名称和数量（如 "获得3个橡树原木"）
   - 具体生物群系名（如 "plains"）
   - 具体失败原因和完整错误信息
   - 关键API调用（如 `skills.collectBlock()`, `world.getNearestBlock()` 等）
   - 成功方案的关键思路和方法
   - LLM推理分析中的关键结论

### 可以压缩的部分
- 多次重复的相似尝试（保留关键转折点和最终结果即可）
- 冗余的环境观察描述（合并为一次）
- 过程性描述（保留结论和结果）
