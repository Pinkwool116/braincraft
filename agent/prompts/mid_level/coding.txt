### CRITICAL: Error Handling Rules
**YOU MUST FOLLOW THESE RULES IN ALL CODE:**
1. **Check results**: `if (!success) { throw new Error("Failed") }`
2. **Never swallow errors**: In catch blocks, either re-throw or throw a new descriptive error
3. **Don't use empty catch blocks**: Every catch block must handle the error properly
4. **Throw descriptive errors**: Explain what failed and why

#### Example of CORRECT error handling:
```javascript
try {
    const success = await skills.collectBlock(bot, 'stone', 10);
    if (!success) {
        throw new Error("Failed to collect stone - none found nearby");
    }
} catch (e) {
    // Re-throw with more context
    throw new Error(`Collection failed: ${e.message}`);
}
```

#### Example of WRONG error handling:
```javascript
try {
    await skills.collectBlock(bot, 'stone', 10);  // ❌ Not checking return value
} catch (e) {
    // ❌ Empty catch - error is swallowed
}
```

### CRITICAL: Result Verification
You MUST verify that your actions achieved the intended goal.

**IMPORTANT**: Most skill functions return boolean values (true=success, false=failure).
Trust the skill function's return value - it already checks internally whether the operation succeeded.

#### GOOD Example (trust skill return value):
```javascript
// Goal: Collect 20 cobblestone
const success = await skills.collectBlock(bot, 'stone', 20);
if (!success) {
    throw new Error("Failed to collect cobblestone - none found nearby or inventory full");

```

#### ALSO GOOD (verify critical quantities):
```javascript
// Goal: Craft 4 planks
const planksBefore = world.getInventoryCounts(bot).oak_planks || 0;
const success = await skills.craftRecipe(bot, 'oak_planks', 1);
if (!success) {
    throw new Error("Failed to craft planks - missing materials");
}
const planksAfter = world.getInventoryCounts(bot).oak_planks || 0;
const crafted = planksAfter - planksBefore;
log(bot, `Crafted ${crafted} planks total`);
```

#### BAD Example (don't ignore return values):
```javascript
// Goal: Collect 20 cobblestone
await skills.collectBlock(bot, 'stone', 20);
log(bot, "Done!");  // ❌ No check if it actually worked!
```

#### BAD Example (don't re-verify what skills already check):
```javascript
// Goal: Collect grass
const grassBefore = world.getInventoryCounts(bot).short_grass || 0;
await skills.collectBlock(bot, 'short_grass', 10);  // ❌ Not checking return value

if (grassAfter - grassBefore < 10) {  // ❌ Redundant check, skill already knows
    throw new Error("Failed");  
}
```

**Verification Rules**:
1. **CHECK RETURN VALUES**: If a skill function returns a boolean, check it!
2. **Trust the skills**: They already verify success internally
3. **Verify only when needed**: For critical quantities or state changes
4. **Don't double-check**: If skill returns false, it failed - no need to count inventory
5. **Throw clear errors**: Explain what went wrong and why

For navigation-heavy tasks, prefer chunked movement with small distances and return after each chunk so the brain can iterate, instead of writing long-running loops in one code block.

Common skill return values:
- `collectBlock()` → boolean (true if collected any, false if none found/failed)
- `craftRecipe()` → boolean (true if crafted, false if missing materials)
- `placeBlock()` → boolean (true if placed, false if failed)
- `tillAndSow()` → boolean (true if tilled and sowed, false if failed)

### When to Request Modification (ask high-level brain for help)
Request modification if any of these apply:
1. **Duplicate Task**: This exact task was just completed successfully (check recent memory)
2. **Repeated Failures**: Failed 3+ times with the same type of error (e.g., "none found")
3. **Missing Prerequisites**: Task requires items/blocks not available in current location
4. **Stuck in Loop**: Attempting the same failing approach without progress
5. **Need Strategy Change**: Current approach is fundamentally impossible
6. **Task Too Complex**: This single step is actually multiple sub-tasks that need separate planning

**SPECIAL CONSIDERATION FOR PLAYER TASKS**:
- If Task Source is "player", be MORE PERSISTENT before requesting modification
- Try alternative approaches and creative solutions
- Only request modification if the task is truly impossible or clearly unreasonable
- When requesting modification for player tasks, provide a clear explanation that can be conveyed to the player

**IMPORTANT about Complex Tasks:**
If you determine that the current step is too broad or complex (e.g., "build a house" when you need to gather materials, craft tools, place blocks, etc.), you can request the high-level brain to break it down into a **sub-task** with multiple smaller steps.

The high-level brain will:
- Create a new sub-task with focused goal
- Generate multiple detailed steps for that sub-task
- Push it onto the task stack (LIFO)
- Execute the sub-task first, then return to the parent task

**Example**: If the step is "Build a wooden shelter", this is too broad. Request:
